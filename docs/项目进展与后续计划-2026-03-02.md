# 项目进展与后续计划（2026-03-02）

## 一、当前已完成工作

### 1. 基础环境与运行
- Flutter 已安装并通过 `flutter doctor -v` 检查（核心能力可用）。
- PostgreSQL 已安装并运行，业务库已创建。
- MinIO（S3 兼容对象存储）已部署并可用。

### 2. 数据库与结构治理
- 完整数据库结构已设计并落地（用户、情侣关系、日常记录、冲突调解、修复任务、周报、记忆映射层）。
- 已支持日常记录媒体拆分：`daily_entries` + `daily_media`。
- 项目结构已规范：
  - `api/sql/` 作为 **Schema 单一事实来源**；
  - `infra/` 仅放部署与环境资产（如 MinIO compose）。

### 3. 后端 API（FastAPI）
已可用接口：
- 系统：`GET /health`
- 日常记录：
  - `POST /daily`（支持文本+media[]）
  - `GET /daily/timeline`
- 冲突调解：
  - `POST /conflict/session`
  - `POST /conflict/input`
  - `POST /conflict/mediate`
- 周报：`GET /weekly/report`（当前为占位版）
- 媒体上传：
  - `POST /media/presign`
  - `POST /media/upload`

### 4. MemOS 接入（MVP）
- `daily` 已接入记忆写入流程：本地 `memory_items` + 外部 MemOS best-effort 同步。
- `conflict`（session/input/mediate）已接入同类流程。
- 已记录同步状态：`pending/synced/failed` 与 `memos_ref_id`。

### 5. 前端（Flutter）
- 首页已改为移动端风格卡片布局，中文显示正常。
- **日常记录页**已可用并联调：可提交记录、查看时间线。
- **冲突调解页**已做可用版 UI 并联调后端流程。
- 每次关键变更均已提交并推送到 GitHub 主分支。

## 二、当前已知问题/限制

1. `weekly/report` 仍是占位逻辑，未做真实聚合计算。
2. 冲突调解输出仍以模板规则为主，未接入更强模型策略。
3. 媒体链路尚未完善：
   - 上传回调校验
   - 媒体元数据自动补全（分辨率、时长等）
   - 资源清理与生命周期策略
4. 前端仍可继续工程化拆分（目前 `main.dart` 承担较多页面与逻辑）。
5. 测试体系未补齐（接口单测/集成测试）。
